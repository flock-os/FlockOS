2 Three-Backend Architecture

"Unless the LORD builds the house, those who build it labor in vain." — Psalm 127:1

FlockOS supports three independent backends, selected at runtime and transparent to all UI code. The same modules call the same API surface regardless of which backend is active.

BackendModeModuleWhen to Use
Google Apps Script sheets (default) TheVine (the_true_vine.js) Default for all churches. Zero cost, zero config beyond GAS deployment. Handles up to ~100 active users.
Firebase / Firestore firebase UpperRoom (the_upper_room.js) Real-time messaging, sub-second reads, scale past GAS quotas. Firestore writes auto-sync back to Sheets via Cloud Functions.
Local / Offline IndexedDB TheWellspring (the_wellspring.js) Offline-first mode. Load a .xlsx backup into IndexedDB, run FlockOS without network. Ideal for mission trips or unreliable internet.

The Backend Toggle

Every data call checks _isFirebaseComms() before deciding which backend to hit:

// Example from the_tabernacle.js
var rows = _isFirebaseComms()
    ? await UpperRoom.listCareCases({ limit: 500 })
    : await TheVine.flock.care.list({ limit: 500 });

// _isFirebaseComms() returns true when:
//   1. UpperRoom module is loaded + initialized
//   2. UpperRoom.getCommsMode() returns 'firebase'
//   Otherwise defaults to 'sheets' (GAS)

You do not set this manually. If the UpperRoom script is loaded and Firebase is configured, it activates automatically. If not, everything falls through to GAS.

GAS vs. Firestore: When to Upgrade

ConstraintGAS (Sheets)Firestore
Read latency1–5 seconds (cold start: 10s+)50–200 ms
Daily reads~50,000 (consumer Gmail)50,000 free / unlimited on Blaze
Script runtime6 min max per callNo limit
Real-time updatesPolling only (30+ seconds)Native real-time listeners
Concurrent users~10–20 before queueing1,000+ concurrent
CostFreeFree tier generous; Blaze = pay-as-you-go

For most churches (under ~100 active users), GAS is perfectly fine. The UI code doesn’t change — the same modules call the same API surface.

6 Authentication & Security

"The name of the LORD is a fortified tower; the righteous run to it and are safe." — Proverbs 18:10

Login Flow

User visits the_wall.html
  → Nehemiah.guard()
      → sessionStorage has valid token?
          YES → redirect to index.html
          NO  → show login form

User submits email + passcode
  → Nehemiah.login(email, passcode)
    → TheVine.auth.login(email, passcode)
      → POST to DATABASE_URL?action=auth.login
        → GAS: hashPasscode(passcode, salt, pepper) — SHA-256
        → Compare hash to AuthUsers tab
        → Check AccessControl: role + status='approved'
        → Return { ok, session: { token, email, role, expires }, profile }
      → sessionStorage.setItem(SESSION_KEY, session)
  → redirect to index.html

Every subsequent API call
  → TheVine auto-appends token + email
    → GAS validates token + expiry
    → requireModule() gates each handler by RBAC level

Session expiry (default 6 hours)
  → API returns { ok: false, code: 'AUTH_EXPIRED' }
  → Nehemiah clears session → redirects to the_wall.html

Password Security

AspectDetail
Hash AlgorithmSHA-256
SchemeSHA-256(pepper + salt + passcode)
Pepper32-char hex string in GAS Script Properties (FLOCK_AUTH_PEPPER). Generated once by initPepper(). Global, never changes.
SaltPer-user 32-char hex in AuthUsers column D. Unique per user.
Hash Storage64-char lowercase hex in AuthUsers column C
Plain TextCleared after migration. Never stored permanently.
MigrationmigrateAllPasswords() upgrades all users from plain text to pepper+salt scheme
🔒 The pepper is irreplaceable. If the pepper is lost, ALL passwords are permanently unrecoverable. Back it up to a password manager immediately after running initPepper().

Session Management

PropertyValue
Storage BackendGAS PropertiesService (server-side) + sessionStorage (client-side)
Key Prefixexp.session.

Registration Flow

  1. User calls auth.register (public) with email, name, passcode
  2. Row created in AuthUsers with status=‘pending’
  3. Email sent to pastors/admins
  4. Admin reviews via users.pending
  5. Admin calls users.approve (sends approval email, creates AccessControl row with role=‘volunteer’) or users.deny

Lockdown Mode

Set LOCKDOWN = TRUE in AppConfig to restrict login to pastor+ roles only. All other users see: “FlockOS is currently in maintenance mode.”

Permission Override System

The permission hierarchy is: DENY > GRANT > Role Default

  1. Check Permissions sheet for DENY override → deny access
  2. Check Permissions sheet for GRANT override → grant access
  3. Fall back to role-based default

This allows a volunteer (level 1) to be granted giving.enter access, or a pastor (level 4) to be denied users.delete capability.

Sync Architecture

Firestore write (create / update / delete)
  → Cloud Function trigger fires
    → Reads settings/sync doc (cached 5 min)
    → POST to GAS: ?action=sync.write&syncSecret=...
      → GAS validates secret, routes to correct sheet tab
        → Row write/update/delete in Google Sheet
          → Sheet stays in sync as single source of truth

Config Document (settings/sync)

Each church’s Firebase project needs a Firestore document at settings/sync containing:

{
  "gasEndpoint": "https://script.google.com/macros/s/.../exec",
  "syncSecret": "<shared-secret>"
}

12 Build System & Automation

A-Build_Churches.sh

The build script at Covenant/Bezalel/Scripts/A-Build_Churches.sh generates all church deployments from the master source. It performs these steps for each church:

  1. rsync — Copy master FlockOS source into Covenant/Nations/<name>/
  2. Database URL — sed replaces the GAS endpoint URL in HTML/JS files
  3. Tagline — Replace default tagline with church-specific tagline
  4. Theme & Background Colors — Inject primary accent color and background
  5. manifest.json — jq updates PWA name, short_name, theme_color
  6. Firebase Config — Inject per-church firebaseConfig object into the_upper_room.js
  7. Truth Editor Local Flag — Inject window.FLOCK_TRUTH_USE_LOCAL = true; before the_truth.js script tag
  8. HTML Titles & Meta — Update title, description, og:tags
  9. Church Brand Name — Replace “FlockOS” references with church name
  10. Per-page updates — the_wall.html, the_pentecost.html, fishing-for-men.html badges/references
  11. Analytics — Inject Google Analytics measurement ID
  12. Favicon & Photos — Update image URLs to per-church assets

17 Deploying a New Church (Step-by-Step)

"Feed my sheep." — John 21:17
🚨 THE GOLDEN RULE: The GAS project gets ONLY Master Code.gs.md pasted as Code.gs. Never create additional files.

Phase 1 — Prerequisites

  • A Google Account (Workspace or personal Gmail)
  • 1 empty Google Sheet (will hold all 200 database tabs)
  • Access to script.google.com
  • The Covenant/Testimony/Architecture/L-Master Code.md file
  • The Covenant/Courts/TheTabernacle/Scripts/ and Covenant/Courts/TheTabernacle/Pages/ folders

Phase 2 — Create the Google Sheet

  1. Create one Google Sheet named FlockOS Database
  2. Copy the Sheet ID from the URL (the long string between /d/ and /edit)

Phase 3 — Create the GAS Project

  1. In the Google Sheet → Extensions → Apps Script
  2. Delete the default Code.gs content
  3. Open Covenant/Testimony/Architecture/L-Master Code.md in VS Code
  4. Copy the entire contents and paste as Code.gs
  5. Save (Ctrl+S)

Phase 4 — Script Properties

Go to ⚙ Project Settings → Script Properties. Add these:

PropertyValueNotes
SHEET_IDYour Google Sheet IDRequired
ADMIN_EMAILadmin@church.comSeed admin email
ADMIN_FIRSTAdminFirst name
ADMIN_LASTUserLast name
ADMIN_PASSWORDtemporary-password⚠ Delete after setup!

Phase 5 — Run Setup

  1. In the GAS editor, select setupFlockOS from the function dropdown
  2. Click ▶ Run
  3. Grant permissions when prompted
  4. Setup creates all tabs, initializes pepper, seeds admin account, runs smoke tests, installs triggers

Phase 6 — Post-Setup Security

⚠ IMMEDIATELY go to Script Properties and delete ADMIN_PASSWORD. Exposed passwords are a security risk.

Phase 7 — Deploy as Web App

  1. Click Deploy → New Deployment
  2. Type: Web App
  3. Execute as: Me
  4. Who has access: Anyone
  5. Click Deploy → copy the deployment URL

Phase 8 — Register the URL

  1. Set CHURCH_APP_URL in Script Properties (paste the deployment URL)
  2. Run registerChurchUrl() in the GAS editor
  3. This writes the URL to AppConfig so emails and TheVine can reference it

Phase 9 — Configure Frontend

  1. Update the_true_vine.js _config block with the new Database URL
  2. OR configure via the Settings → Provisioning page in the admin dashboard
  3. Test: open [DATABASE_URL]?action=health — should return { ok: true }

Phase 10 — Optional Firebase Setup

  1. Create a Firebase project (e.g., flockos-churchname)
  2. Enable Firestore Database
  3. Enable Authentication (anonymous + email/password)
  4. Add web app in Firebase Console → copy firebaseConfig object
  5. Add to Covenant/Scrolls/ChurchRegistry/ChurchName.json
  6. Run setupFirestoreSync() from GAS to create settings/sync Firestore doc
  7. Deploy Cloud Functions: firebase use <project-id> && firebase deploy --only functions

Phase 11 — Add to Build Script

  1. Add the church’s config block to Covenant/Scrolls/ChurchRegistry/ChurchName.json
  2. Run bash Covenant/Bezalel/Scripts/A-Build_Churches.sh to generate the deployment
  3. Commit and push to GitHub Pages

Verification Checklist

  • Health check: ?action=health returns ok: true
  • Login: seed admin can log in at the_wall.html
  • Dashboard loads with greeting and sidebar
  • Members tab shows empty list (no errors)
  • Notification bell shows (may be empty)
  • Service Worker caches all assets on first load
  • PWA “Add to Home Screen” works on iOS/Android

🚨 If something isn’t working, count your files. The GAS project should have exactly 1 file (Code.gs). If you have more, delete the extras.

19 Maintenance & Operations

"Be diligent to present yourself approved to God, a worker who has no need to be ashamed." — 2 Timothy 2:15

Add a New User (Manual)

  1. Open Google Sheet → AuthUsers tab → add row with email, temporary passcode, role, status=active
  2. Add matching row to AccessControl tab (email, role, name, group, status=approved)
  3. In GAS editor: run migrateAllPasswords to hash the passcode
  4. Give user their email + passcode → they log in at the_wall.html

Or use the admin dashboard: My Flock Portal → Add Member → enable “Create Login Account” section.

Change a User’s Role

  1. Update AccessControl tab → role column
  2. Also update AuthUsers tab → role column to keep in sync
  3. Change takes effect on next login (or session expiry)

Disable a User

  1. AuthUsers: set status to suspended
  2. AccessControl: set status to suspended
  3. AuthUsers: clear token column to kill active session

Update Deployed Backend Code

  1. Edit Master Code.gs.md in VS Code
  2. Open the GAS project at script.google.com
  3. Select all in Code.gs → paste updated contents
  4. Deploy → Manage deployments → ✏️ pencil → New version → Deploy
